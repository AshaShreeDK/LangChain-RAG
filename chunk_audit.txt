Customer data must be retained only for the duration required by business and regulatory needs.
Data retention periods must be documented and approved.

Expired data must be securely deleted.
Backups must follow the same data retention requirements.

Exceptions to retention policies require formal approval.

Temporary access must have an expiration date.

All production access must be logged and monitored.
Shared accounts are strictly prohibited.

Production access reviews must be conducted quarterly.

--- Chunk 3 ---
All changes to production systems must follow the change management process.
Emergency changes must be documented and reviewed after implementation.

Change requests must include risk assessment and rollback plans.
Changes impacting customers must be communicated in advance where possible.

Unauthorized changes to production systems are not permitted.

EC2 instances used for production workloads must be launched only within approved VPCs.
Public IP addresses are not permitted unless explicitly approved by the security team.

All EC2 instances must use IAM roles for AWS access.
The use of long-term access keys on EC2 instances is strictly prohibited.

--- Chunk 6 ---
All code changes must go through a pull request review process.
Direct commits to the main branch are not allowed.

CI pipelines must include unit tests and static code analysis.
Builds that fail quality checks must not be promoted to deployment stages.

Production deployments require manual approval.
Rollback procedures must be documented for every service.

Docker images must be built using minimal base images.
Alpine-based images are preferred where possible.

Containers must not run as the root user.
Secrets must not be stored inside Docker images.

All images must be scanned for vulnerabilities before deployment.
Only approved container registries may be used in production.

Pods without resource limits are not permitted in production namespaces.

Liveness and readiness probes must be configured for all services.
Deployments must use rolling update strategies to avoid downtime.

Secrets must be managed using Kubernetes Secrets or external secret managers.

--- Chunk 9 ---
A Sev-1 incident is defined as a complete service outage impacting customers.
The on-call engineer must acknowledge a Sev-1 alert within 5 minutes.

During active incidents, status updates must be communicated every 30 minutes.
Incident mitigation should focus on service restoration before root cause analysis.

A post-incident review must be conducted within 5 business days.
{'source': 'incident_response_runbook.txt', 'path': 'sre/incidents/incident_response_runbook.txt', 'domain': 'sre'}

Service Level Agreements (SLAs) must align with defined SLOs.

Error budgets must be tracked and reviewed regularly.

SLA breaches must be reported to stakeholders promptly.
